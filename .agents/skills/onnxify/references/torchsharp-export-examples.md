# TorchSharp Export Examples

## Purpose

Use this page when the task is not just "which `Export(...)` overload exists?" but "how should a TorchSharp model be structured so its ONNX export succeeds, and what should the model's custom `Export()` method actually look like?"

This reference is based on the real examples in `src/Onnxify.Examples` and is meant to complement, not replace, the autogenerated converter catalog in `references/torchsharp-converters/index.md`.

## Core Rule

Treat `forward(...)` and `Export(...)` as two views of the same computation:

- `forward(...)` runs tensors through TorchSharp modules and tensor ops.
- `Export(...)` recreates the same logical stages as ONNX graph edges and ONNX nodes.

The closer the stage boundaries match, the higher the chance that the exported graph preserves the intended semantics.

## What Usually Works Best

For `Onnxify.TorchSharp`, the most reliable model shape is:

1. Keep reusable feed-forward parts as registered TorchSharp modules such as `Sequential`, `Conv*`, `BatchNorm*`, `Linear`, `Embedding`, `LayerNorm`, `LSTM`, and activations.
2. In `Export(...)`, reuse those same modules through `module.Export(graph, input)` whenever the computation is still a straightforward module-to-module chain.
3. Re-express tensor-only operations explicitly with `OnnxGraph` when `forward(...)` uses shape changes, tuple outputs, residual adds, masks, tied weights, or custom attention math.
4. Make the ONNX input/output contract explicit through `graph.AddInput(...)`, `graph.AddOutput(...)`, and a final named edge, usually via `Identity`.

## Important Constraint: What `Sequential.Export(...)` Assumes

`src/Onnxify.TorchSharp/TorchModuleExtensions.cs` implements `Sequential.Export(...)` as a simple walk over registered child modules in order.

That works well when the TorchSharp code is effectively:

```csharp
var x = block1.forward(input);
x = block2.forward(x);
x = block3.forward(x);
return x;
```

It is not enough by itself when `forward(...)` contains:

- `view`, `reshape`, `permute`, `transpose`, or `flatten` outside registered child modules
- tuple outputs like `var (y, h, c) = lstm.forward(x)`
- residual or skip connections like `x = x + branch.forward(x)`
- manual tensor math such as masks, tied projections, or custom attention score computation

In those cases, keep using child-module exporters where possible, but bridge the gaps with direct `OnnxGraph` ops.

## Example 1: Pure Sequential Stages

Best first file: `src/Onnxify.Examples/Models/TorchSharpExportShowcase.cs`

This example is the clearest baseline because both the feature extractor and classifier are plain `Sequential` chains:

```csharp
public override Tensor forward(Tensor input)
{
    var x = _features.forward(input);
    x = _classifier.forward(x);
    return x;
}

public OnnxModel Export()
{
    var model = OnnxModel.Create(new OnnxModelCreationOptions { Opset = 22 });
    var graph = model.Graph;

    var input = graph.AddInput(
        name: "input",
        type: OnnxTensorType.Create<float>(["batch", 3, 16, 16])
    );

    var x = _features.Export(graph, input);
    x = _classifier.Export(graph, x);

    var outputEdge = graph.AddEdge("output");
    graph.Identity(
        name: "output_identity",
        options: new IdentityInputOutputOptions
        {
            Input = x,
            Output = outputEdge,
        }
    );

    graph.AddOutput(
        name: "output",
        type: OnnxTensorType.Create<float>(["batch", _numClasses])
    );

    return model;
}
```

Why this pattern is robust:

- The model is decomposed into named stages with clear ownership.
- Each stage is expressible as registered TorchSharp modules that already have exporters.
- The ONNX graph shape is obvious: one input, one linear chain, one output.

Use this pattern when the model is mostly "feature stack -> classifier stack" and the only glue is sequential composition.

## Example 2: Shape Logic Must Become Graph Logic

Best files:

- `src/Onnxify.Examples/Models/AlexNet.cs`
- `src/Onnxify.Examples/Models/TinyYoloLikeDetector.cs`

### AlexNet: `view(...)` in `forward(...)` becomes `Flatten(...)` in ONNX

`AlexNet.forward(...)` contains a shape operation between `_features` and `_classifier`:

```csharp
public override Tensor forward(Tensor input)
{
    var x = _features.forward(input);
    x = x.view([x.shape[0], 256 * 6 * 6]);
    x = _classifier.forward(x);
    return x;
}
```

Its `Export()` mirrors that exact boundary:

```csharp
var x = _features.Export(graph, input);

x = graph.Flatten(
    name: "flatten",
    options: new FlattenInputOptions
    {
        Input = x,
        Axis = 1,
    }
);

x = _classifier.Export(graph, x);
```

The key lesson is that `view(...)` is not a side note. If it changes the semantic interface between major blocks, it needs a corresponding ONNX op in `Export()`.

### Tiny YOLO-like detector: `view + permute + view` becomes `Reshape + Transpose + Reshape`

`TinyYoloLikeDetector.forward(...)` transforms raw convolution output into detector predictions:

```csharp
x = x
    .view([batch, _anchorCount, _attributeCount, _gridHeight, _gridWidth])
    .permute(0, 1, 3, 4, 2)
    .contiguous()
    .view([batch, PredictionCount, _attributeCount]);
```

The export method rewrites that into ONNX-native graph ops:

```csharp
var reshapeToGridShape = graph.AddTensor(
    name: "detector_grid_shape",
    shape: [5],
    value: [0L, _anchorCount, _attributeCount, _gridHeight, _gridWidth]
);

x = graph.Reshape(...);
x = graph.Transpose(... Perm = [0, 1, 3, 4, 2]);

var reshapeToPredictionsShape = graph.AddTensor(
    name: "detector_predictions_shape",
    shape: [3],
    value: [0L, PredictionCount, _attributeCount]
);

x = graph.Reshape(...);
```

Use this pattern whenever `forward(...)` changes layout or rank after an exported module chain. Keep the same logical stages; only the implementation surface changes from Torch tensor ops to ONNX graph ops.

## Example 3: Recurrent Models With Tuple Outputs

Best file: `src/Onnxify.Examples/Models/LSTMLIDModel.cs`

`LSTMLIDModel` is a strong example of "mostly supported modules, plus one manual reduction":

```csharp
public override Tensor forward(Tensor input)
{
    var x = _charEmbeddings.forward(input);
    var (lstm, _, _) = _lstm.forward(x);
    var linear = _hidden2Lang.forward(lstm);
    var logit = torch.sum(linear, 1);
    return logit;
}
```

The export stays close to that structure:

```csharp
var x = _charEmbeddings.Export(graph, input);
x = _lstm.Export(graph, x).Y ?? throw new Exception();
x = _hidden2Lang.Export(graph, x);

x = graph.ReduceSum(
    name: "sum_logits",
    options: new ReduceSumInputOptions
    {
        Data = x,
        Axes = graph.AddTensor<long>("sum_logits_axes", [1], [1]),
        Keepdims = 0,
    }
);
```

Why this pattern matters:

- `Embedding` maps cleanly to `Embedding.Export(...)`.
- `LSTM` does not return a single edge. It returns `LSTMOutput` with `Y`, `YH`, and `YC`.
- The model-level export explicitly chooses the output that matches the Torch forward path, here `Y`.
- The final sequence reduction remains model-specific, so it is written directly with `graph.ReduceSum(...)`.

Useful detail from `TorchModuleExtensions.cs`:

- the shared LSTM exporter already handles `batch_first=true` by transposing into ONNX's expected layout and back again
- it also handles ONNX-vs-PyTorch gate ordering internally

That means the model-level `Export()` should focus on which LSTM output is semantically needed, not on reimplementing LSTM internals.

## Example 4: Embeddings, Constants, And Graph-Owned Data

Best file: `src/Onnxify.Examples/Models/MiniGpt2LikeModel.cs`

Transformer-style exports usually need graph-owned constants in addition to exported modules. In `MiniGpt2LikeModel.Export()`, the graph creates:

- token id input
- constant position ids
- constant causal mask

```csharp
var input = graph.AddInput(
    name: "input",
    type: OnnxTensorType.Create<long>(["batch", _settings.MaxContextLength])
);

var positionIds = graph.AddTensor(
    name: "position_ids",
    shape: [1, _settings.MaxContextLength],
    value: Enumerable.Range(0, _settings.MaxContextLength).Select(static x => (long)x).ToArray()
);

var causalMask = graph.AddTensor(
    name: "causal_mask",
    shape: [1, 1, _settings.MaxContextLength, _settings.MaxContextLength],
    value: CreateAdditiveCausalMask(_settings.MaxContextLength)
);
```

Then the model exports both embeddings and manually combines them:

```csharp
var x = _tokenEmbedding.Export(graph, input);
var positions = _positionEmbedding.Export(graph, positionIds);

x = graph.Add(
    name: "token_plus_position",
    options: new AddInputOptions
    {
        A = x,
        B = positions,
    }
);
```

This is a strong pattern whenever the Torch forward pass depends on deterministic tensors that are better represented as ONNX initializers or constants than as runtime inputs.

## Example 5: Custom Composite Blocks Should Get Their Own `Export(...)`

Best file: `src/Onnxify.Examples/Models/MiniGpt2LikeModel.cs`

The transformer block does not try to force everything through one giant shared exporter. Instead, each composite module gets a model-local export method that mirrors its `forward(...)`.

TorchSharp:

```csharp
public override Tensor forward(Tensor x)
{
    x = x + _attention.forward(_attentionNorm.forward(x));
    x = x + _feedForward.forward(_feedForwardNorm.forward(x));
    return x;
}
```

ONNX export:

```csharp
var attentionNorm = _attentionNorm.Export(graph, input);
var attention = _attention.Export(graph, attentionNorm, causalMask);

var residual = graph.Add(
    name: graph.NextName("attn_residual"),
    options: new AddInputOptions
    {
        A = input,
        B = attention,
    }
);

var feedForwardNorm = _feedForwardNorm.Export(graph, residual);
var mlp = _feedForward.Export(graph, feedForwardNorm);

return graph.Add(
    name: graph.NextName("mlp_residual"),
    options: new AddInputOptions
    {
        A = residual,
        B = mlp,
    }
);
```

This is the most important design pattern in the examples:

- create a custom `Export(...)` on the composite module
- keep the same stage boundaries as `forward(...)`
- reuse shared exporters for the leaves
- express residual joins and other tensor algebra explicitly with `OnnxGraph`

If a block has a meaningful architectural identity in TorchSharp, it often deserves a matching export helper instead of flattening all its logic into one huge top-level `Export()`.

## Example 6: Fused Attention Often Needs Manual ONNX Composition

Best file: `src/Onnxify.Examples/Models/MiniGpt2LikeModel.cs`

`MiniGpt2CausalSelfAttention` shows where shared module exporters stop being enough. The Torch code performs:

- one fused `Linear` for QKV
- reshape into heads
- transpose
- score matmul
- scaling
- causal mask add
- softmax
- value matmul
- merge heads
- output projection

The export method mirrors those stages directly in ONNX by:

- slicing rows out of the fused linear weights for query, key, and value
- creating ONNX weight tensors from Torch parameters
- using `Reshape`, `Transpose`, `MatMul`, `Mul`, `Add`, and `Softmax`
- merging heads back before the final projection

This is the right approach when the model's behavior is a composite tensor program rather than a single supported Torch module.

Do not wait for the shared export layer to understand every custom attention pattern automatically. If the model's architecture is specific, write a local export helper that makes the ONNX graph structure explicit.

## How To Decide Between Shared Exporters And Local Graph Code

Keep the logic in `Onnxify.TorchSharp/TorchModuleExtensions.cs` when:

- the behavior belongs to a reusable TorchSharp leaf module type
- multiple models would benefit from the same conversion rule
- the mapping is fundamentally "this module type maps to these ONNX ops"

Keep the logic local to the example or owning model when:

- the graph composition is model-specific
- the forward path mixes supported modules with custom tensor algebra
- the model needs architecture-specific constants, masks, or metadata
- the export depends on tied weights or custom output shaping

## Export Design Checklist

When writing a new TorchSharp model for export, maximize your success rate by checking:

- Does `forward(...)` decompose into named submodules with stable boundaries?
- Can each simple leaf remain a real TorchSharp module instead of disappearing into ad hoc tensor code?
- Does every shape/layout transition in `forward(...)` have an ONNX equivalent in `Export()`?
- If a module returns multiple outputs, did you explicitly choose the correct exported member?
- If the model uses residuals or branches, did you represent those joins explicitly in the ONNX graph?
- If the model depends on deterministic tensors such as masks, position ids, or fixed shapes, did you decide whether they belong as graph constants or runtime inputs?
- Did you keep model-specific graph composition local instead of bloating the shared converter layer?
- Did you add metadata when downstream code needs vocabulary, label, or layout hints?

## Practical Heuristic

If you are unsure how to export a new model, start by writing `forward(...)` in a way that naturally decomposes into:

- reusable exported modules for the obvious leaf operations
- explicit model-local helper methods for the topology-specific tensor logic

That split matches the current strengths of `Onnxify.TorchSharp` and gives the clearest path to a faithful ONNX graph.
