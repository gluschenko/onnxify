# Onnxify API Surface

Use this reference when you need concrete entry points for repository work in the core Onnxify library or the TorchSharp export layer.

## Core Types

- `src/Onnxify/OnnxModel.cs`
  - `OnnxModel.Create(...)`
  - default creation profile: standard ONNX opset 25 and IR version 11
  - ONNX baseline: opset 25 from bundled standard-domain operator schemas; IR version 11 as the repository compatibility target for generated models
  - `OnnxModel.FromFile(path)`
  - `model.Save(path, overwrite)`
  - model metadata: `ProducerName`, `ProducerVersion`, `ModelVersion`, `IrVersion`, `Domain`, `Document`
  - `model.Graph`

- `src/Onnxify/OnnxGraph.cs`
  - collections: `Inputs`, `Outputs`, `Initializers`, `IntermediateValues`, `Nodes`
  - lookups: `GetNode(name)`, `GetValue(name)`
  - builders: `AddInput`, `AddOutput`, `AddValue`, `AddTensor`, `AddEdge`, `AddNode`
  - input/output markers: `AddInput(OnnxValue)`, `AddOutput(OnnxValue)`, `RemoveInput(...)`, `RemoveOutput(...)`
  - graph editing: `RemoveNode(...)`, `ReplaceNode(...)`, `RemoveValue(...)`, `ReplaceValue(...)`, `RemoveTensor(...)`, `RemoveEdge(...)`
  - removal behavior: value, tensor, and edge removal clears matching node input/output references; node removal prunes unused loose edges
  - naming: `NextName(prefix)`

- Other common types
  - `src/Onnxify/OnnxNode.cs`
  - `src/Onnxify/OnnxTensor.cs`
  - `src/Onnxify/OnnxValue.cs`
  - `src/Onnxify/OnnxValueType.cs`
  - `src/Onnxify/OnnxAttribute.cs`

## TorchSharp Export Surface

- Dependency baseline: `Onnxify.TorchSharp` references `TorchSharp` `0.106.0`; runtime package examples should use matching `TorchSharp-cpu` or TorchSharp CUDA runtime versions.

- `src/Onnxify.TorchSharp/TorchModuleExtensions.cs`
  - dispatch entry point: `TorchModule.Export(graph, input)`
  - concrete exporters for `Sequential`, `Conv*`, `BatchNorm*`, activations, pooling, `Flatten`, `Embedding`, `LayerNorm`, `Upsample`, `Linear`, `LSTM`, and more
  - translation rules for TorchSharp-specific semantics such as padding expansion, resize options, batch-first handling, and LSTM gate order

- `src/Onnxify.TorchSharp/TorchHelper.cs`
  - tensor shape extraction: `GetShape(...)`
  - tensor data extraction: `GetFloatData(...)`
  - option normalization: `ToLongArray(...)`, `ExpandPadding(...)`

- `src/Onnxify.TorchSharp/TorchOpAttribute.cs`
  - attaches Torch operator names to export methods
  - useful when extending or auditing coverage

- `src/Onnxify.TorchSharp.Observer/torchsharp-operator-report.md`
  - operator coverage snapshot for TorchSharp-related surfaces
  - useful when checking whether a module or operator pattern is already represented

- `src/Onnxify.AgentSkillGenerator/Program.cs`
  - entry point for deterministic regeneration of skill docs inside the agent skill
  - rewrites the generated files under `references/operators` and `references/torchsharp-converters`

- `references/operators/index.md`
  - autogenerated operator catalog for the skill
  - links to one markdown file per reflected Onnxify operator with schema descriptions, wrapper signatures, and TorchSharp converter coverage

- `references/torchsharp-converters/index.md`
  - autogenerated catalog of public `Onnxify.TorchSharp` `Export(...)` signatures
  - links to one markdown file per public converter overload, including receiver type, return shape, and attached Torch op names

## Model Generator Surface

- `src/Onnxify.ModelGenerator/OnnxModelGenerator.cs`
  - incremental source generator that reads ONNX model files and emits typed `OnnxRuntime` wrappers
  - generated wrappers expose `Inputs`, `Outputs`, `OutputNames`, and multiple `Run(...)` overloads
  - supports per-file naming overrides through `OnnxifyModelClassName` and `OnnxifyModelNamespace`

- `src/Onnxify.ModelGenerator/Onnxify.ModelGenerator.props`
  - defines the `OnnxModel` MSBuild item
  - supports opt-in default discovery through `EnableDefaultOnnxModelItems`

- `src/Onnxify.ModelGenerator/Onnxify.ModelGenerator.targets`
  - forwards `OnnxModel` items into `AdditionalFiles`
  - preserves output-copy metadata for deployed `.onnx` assets

- `src/Onnxify.Tests/OnnxModelGeneratorTests.cs`
  - source of truth for generated wrapper shape, optional inputs, metadata overrides, and diagnostic behavior

- `references/model-generator.md`
  - practical guide for when and why to use `Onnxify.ModelGenerator`
  - includes `csproj` setup and runtime code examples

## ML Pipeline Surface

- `src/Onnxify.ML/Pipeline.cs`
  - `Pipeline.Begin<TInput>()`
  - `Pipeline<TInput, TOutput>.ExecuteAsync(...)`
  - `pipeline.RunToListAsync(...)`
  - aggregate progress calculation through `CalculateProgress(...)`

- `src/Onnxify.ML/PipelineStage.cs`
  - base abstractions for custom stages
  - `PipelineStage<TInput, TOutput>.Then(...)`
  - `PipelineStage<TInput, TOutput>.Build()`

- `src/Onnxify.ML/PipelineContext.cs`
  - execution-local property bag and typed state
  - service-provider access
  - sequence numbers and progress callback plumbing

- `src/Onnxify.ML/Stages`
  - item-wise stages: `MapStage`, `FilterStage`, `TapStage`, `DelegateStage`
  - batching stages: `MiniBatchStage`, `BatchingStage<TInput, TBatch>`
  - orchestration stages: `EpochStage`, `BranchStage`, `ForkStage`
  - concurrency stage: `ParallelMapStage`

- `src/Onnxify.ML.TorchSharp/Stages`
  - `TorchTensorBatchStage<TSample>` for collate + tensor materialization
  - `TorchInferenceStage<TBatch, TModelOutput, TResult>` for forward-only execution
  - `TorchTrainingStage<TBatch, TModelOutput, TResult>` for forward/loss/backward/optimizer-step execution

- `references/ml-pipelines.md`
  - practical guide for when and why to use `Onnxify.ML` and `Onnxify.ML.TorchSharp`
  - includes inference and training pipeline examples

## Hugging Face Surface

- `src/Onnxify.HuggingFace/HuggingFaceClient.cs`
  - `DownloadRepositoryAsync(repositoryId, targetDirectoryPath, options, progress, cancellationToken)`
  - downloads Hugging Face repository files into a local directory while preserving repository-relative paths
  - supports injected `HttpClient` for testing or custom endpoints

- `src/Onnxify.HuggingFace/HuggingFaceDownloadOptions.cs`
  - `Revision`
  - `AccessToken`
  - `IncludePath` and `ExcludePath` path filters
  - `ProgressCallback`
  - `Overwrite`
  - `BufferSize`

- `src/Onnxify.Tests/HuggingFaceClientTests.cs`
  - source of truth for filtered downloads, progress callbacks, and path-safety behavior

## Best Existing Examples

  - creation options

- `src/Onnxify.Tests/OnnxProjectGeneratorTests.cs`

  - asserting emitted source and asset files

  - end-to-end expectations for generated `OnnxRuntime` wrapper classes
  - good first stop when you need the exact public shape emitted by `Onnxify.ModelGenerator`

- `src/Onnxify.ConsoleTest/Program.cs`
  - playground for loading assets and composing graphs
  - examples of `Conv`, `Relu`, `MaxPool`, `Flatten`, `Gemm`

- `src/Onnxify.Examples/Program.cs`
  - end-to-end example around exporting and evaluating a model

- `src/Onnxify.Examples/Models/TorchSharpExportShowcase.cs`
  - compact end-to-end example of exporting a `Sequential`-heavy TorchSharp model into ONNX
  - good first stop for common export conventions

- `src/Onnxify.Examples/Models/LSTMLIDModel.cs`
  - example of `Embedding` + `LSTM` + `Linear` export
  - shows how TorchSharp export is composed with manual ONNX graph ops like `ReduceSum` and output wiring

- `src/Onnxify.Examples/Models/MiniGpt2LikeModel.cs`
  - advanced mixed approach: reusable TorchSharp exports plus custom ONNX graph composition
  - useful when a model needs higher-level graph logic beyond simple module-to-node translation

- `src/Onnxify.Examples/Models/AlexNetTrainer.cs`
  - the main end-to-end `Onnxify.ML` + `Onnxify.ML.TorchSharp` training pipeline example

- `src/Onnxify.Tests/OnnxifyPipelineTests.cs`
  - smallest focused examples of `MapStage`, `MiniBatchStage`, `ParallelMapStage`, `BranchStage`, `ForkStage`, `TorchInferenceStage`, and `TorchTrainingStage`
  - good first stop when the user wants a minimal pipeline shape rather than a full example app

## Useful Heuristics

- If behavior is about raw ONNX persistence, start in the library and tests, not in `ConsoleTest`.
- If the repo already has a typed operator helper, copy that pattern instead of emitting raw stringly-typed nodes.
- If the task is operator-specific, use `references/operators/index.md` to find the generated operator doc before reading source files one by one.
- If the task is about TorchSharp `Export(...)` overloads themselves, use `references/torchsharp-converters/index.md` before scanning `TorchModuleExtensions.cs` manually.
- If behavior is about translating a TorchSharp module into ONNX, start in `TorchModuleExtensions.cs`, not in the example model.
- If the change is model-specific composition rather than generic module export, keep it in the example or owning project instead of bloating the shared export layer.
- If the export path depends on inference-only semantics, prefer a clear guard over a silent approximation.
- If the task is about runtime training or inference workflow composition rather than ONNX graph export, start in `references/ml-pipelines.md` and the `src/Onnxify.ML*` projects instead of `Onnxify.TorchSharp`.
- If you changed operator wrappers, ONNX schema inputs, or TorchSharp converter coverage, regenerate the skill docs through `src/Onnxify.AgentSkillGenerator`.
- If adding a new helper, confirm whether the task belongs in the main library or a placeholder project.
- For regression-proofing, prefer assertions on names, shapes, attributes, and edge wiring over only checking counts.
